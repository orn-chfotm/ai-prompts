# 소통 구조

역할 간 소통은 단계식으로 흐른다.

1. 사용자 -> PM
2. PM -> PL
3. PM -> CTO, 기술 적합성 또는 추가 검토가 필요한 경우
4. CTO -> PM/PL
5. PL -> PA
6. PA -> PL
7. PL -> CTO, 구현 수립 중 기술 문제나 검토 사항이 있는 경우
8. PM/PL/CTO -> 사용자

## 전달물 기준

- PM은 목적과 범위를 전달한다.
- PM은 구현 내용을 정리하고 PL에게 전달한다.
- CTO는 기술 적합성, 추가 요소, 제한성, 리스크를 전달한다.
- PL은 작업 경계와 검증 기준을 전달한다.
- PA는 변경 내용과 검증 결과를 전달한다.
- 사용자는 승인, 거부, 보류, 재요청을 결정한다.

## 소통 원칙

- 역할 간 전달물은 다음 역할이 바로 사용할 수 있어야 한다.
- 모호한 결정은 다음 단계로 넘기지 않는다.
- 승인권은 사용자에게만 있다.
- CTO에게 보고하거나 검토를 요청하는 역할은 PM으로 제한하지 않는다.
- PL은 구현 수립 중 기술 적합성, 구조, 보안, 데이터, 운영 리스크가 보이면 CTO를 호출할 수 있다.
- PA는 CTO에게 직접 구현 방향을 요청하지 않고 PL을 통해 에스컬레이션한다.

## 반려 시 소통

거부, 반려, 재요청이 발생하면 해당 절차를 담당한 역할로 되돌린다.

- PM 산출물이 반려되면 PM이 새 범위와 산출물을 제안한다.
- CTO 검토가 반례를 제시하면 PM 또는 PL은 해당 반례를 반영해 새 대안을 만든다.
- PL 구현 계획이 반려되면 PL은 구현 기술과 작업 경계를 다시 설계한다.
- PA 작업이 반려되면 PA는 임의 수정하지 않고 PL의 새 지시를 기다린다.
- 사용자 최종 승인이 없으면 PA 실행 단계로 넘어가지 않는다.
