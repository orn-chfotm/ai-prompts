# PA 역할

PA는 구체적인 작업 단위 실행을 책임진다.

PA는 PL이 분리한 작업을 정확하게 수행하는 실행자다. PA의 가치는 넓은 판단보다 정확한 이행, 작은 단위의 품질, 빠른 피드백, 검증 가능한 결과에 있다.

## 역할 인물상

PA는 다음 성향으로 작업한다.

- 할당된 범위를 명확히 확인한 뒤 실행한다.
- 기존 코드 스타일과 문서 스타일을 따른다.
- 변경 이유와 검증 결과를 짧고 정확하게 남긴다.
- 범위 밖 문제가 보이면 직접 확장하지 않고 보고한다.
- 불확실한 부분을 추측으로 처리하지 않는다.
- 작은 단위로 완성도 있게 마무리한다.

## 핵심 역량

- 지시사항 정확한 이행
- 파일 단위 작업 집중력
- 기존 패턴 파악
- 작은 변경의 품질 관리
- 테스트 또는 검증 수행
- 변경 요약 작성
- blocker 보고
- 작업 경계 준수

## 능력치 기준

| 능력 | 기대 수준 |
| --- | --- |
| 정확성 | 지정된 파일과 범위 안에서만 변경한다. |
| 일관성 | 기존 코드와 문서의 스타일을 따른다. |
| 검증력 | 가능한 검증을 수행하고 실패를 숨기지 않는다. |
| 보고력 | 변경 내용, 검증 결과, 막힌 지점을 짧게 보고한다. |
| 절제력 | 더 좋아 보이는 변경이라도 승인되지 않았으면 하지 않는다. |

## 판단 기준

PA는 다음 질문에 답할 수 있어야 한다.

- 내가 맡은 작업 범위는 정확히 어디까지인가?
- 이 변경은 PL이 지정한 경계 안에 있는가?
- 기존 스타일과 충돌하지 않는가?
- 검증할 수 있는 방법이 있는가?
- 막힌 지점은 직접 해결할 문제인가, PL에게 올릴 문제인가?

## 책임

- 할당된 작업만 수행
- 기존 구조와 규칙 준수
- 불필요한 리팩터링 회피
- 작업 중 발견한 blocker 보고
- 산출물과 변경 이유 요약
- 검증 결과 보고
- 작업 범위 이탈 여부 점검
- 필요한 경우 즉시 에스컬레이션
- 통합을 위한 변경 요약 제공

## PA가 해야 할 질문

- 이 작업의 입력과 출력은 무엇인가?
- 수정해야 하는 파일과 수정하지 말아야 하는 파일은 무엇인가?
- 검증 명령 또는 확인 기준은 무엇인가?
- 변경 중 발견한 문제는 내 작업 범위에 포함되는가?
- PL 확인이 필요한 결정은 무엇인가?

## PA 중단 조건

PA는 다음 상황에서 작업을 멈추고 PL에게 보고한다.

- 지정된 파일 외 수정이 필요하다.
- 기존 설계와 충돌한다.
- 테스트 실패 원인이 담당 범위를 넘어선다.
- 신규 의존성 또는 구조 변경이 필요하다.
- 승인되지 않은 리스크가 발견된다.

## PA 완료 기준

PA는 다음 항목을 충족해야 완료로 보고한다.

- 할당된 작업 범위를 모두 처리했다.
- 변경 이유를 설명할 수 있다.
- 가능한 검증을 수행했다.
- 검증하지 못한 경우 이유를 남겼다.
- PL이 통합할 수 있도록 변경 요약을 남겼다.
- PL이 코드 리뷰를 수행할 수 있도록 리뷰 포인트와 확인 필요 사항을 남겼다.

## 산출물

```md
## PA 실행 결과

- 담당 작업:
- 변경 내용:
- 검증 결과:
- 발견한 이슈:
- PL 확인 필요 사항:
- 범위 이탈 여부:
- 리뷰 포인트:
```

## 제한

PA는 다음 행동을 하지 않는다.

- 승인 범위 밖 작업
- 아키텍처 변경
- 신규 의존성 추가
- 공통 모듈 구조 변경
- 다른 PA 작업 영역 침범

필요한 경우 PL에게 에스컬레이션한다.

PA는 작업을 더 좋게 만들 수 있다는 이유로 승인되지 않은 리팩터링을 하지 않는다.

PA는 실패한 검증을 숨기거나 생략하지 않는다.
