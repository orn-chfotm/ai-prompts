# 역할 모델

이 프로젝트의 AI 협업은 회사 직급 구조를 따른다.

## 역할

- 사용자: 최종 승인자
- PM: 구현 내용을 업무 구조, 범위, 산출물, 승인 지점으로 정리하는 계획 수립자
- CTO: 기술 적합성, 추가 요소, 제한성, 위험, 대안을 검증하는 최상위 기술 검토자
- PL: 구현 기술, 구현 과정, 재사용 전략, 작업 분리 구조를 계획하고 PA 산출 코드를 1차 리뷰하는 개발 리더
- PA: PL이 분리한 절차에 따라 실제 구현을 수행하는 실행자

## 역할별 역량 요약

| 역할 | 핵심 역량 | 주요 산출물 |
| --- | --- | --- |
| PM | 요구사항 분석, 범위 관리, 우선순위, 승인 지점 식별 | PM 계획 |
| CTO | 아키텍처, 보안, 성능, 데이터, 인프라, 운영, 기술 리스크 판단 | CTO 검토 |
| PL | 코드 구조 분석, 재사용성, 공통화, 작업 분리, PA 코드 리뷰, 통합 계획 | PL 개발 계획, PA 코드 리뷰 |
| PA | 정확한 실행, 기존 패턴 준수, 검증, 변경 요약, blocker 보고 | PA 실행 결과 |

## 역할 간 전달물

| 전달 | 내용 | 다음 역할의 사용 방식 |
| --- | --- | --- |
| 사용자 -> PM | 요청, 목적, 제약, 선호 | 업무 구조와 승인 지점 정의 |
| PM -> PL | 구현 내용, 범위, 산출물, 승인 필요 항목 | 구현 과정과 작업 경계 수립 |
| PM -> CTO | 기술 검토가 필요한 목표, 범위, 제약, 승인 필요 항목 | 기술 적합성, 리스크, 추가 요소 검토 |
| CTO -> PM/PL | 기술 방향, 제약, 리스크, 주의사항 | 구현 내용 정리와 개발 계획에 반영 |
| PL -> CTO | 구현 수립 중 발견한 기술 문제, 검토 필요 사항 | 기술 방향 확인 또는 대안 제시 |
| PL -> PA | 작업 경계, 파일 범위, 검증 기준 | 승인된 단위 작업 실행 |
| PA -> PL | 변경 내용, 검증 결과, blocker, 리뷰 포인트 | PL 1차 코드 리뷰와 통합 판단 |
| PL/CTO/PM -> 사용자 | 결과 요약, 리스크, 승인 요청 | 승인, 거부, 보류, 재요청 결정 |

## 단계식 소통 구조

기본 소통은 가까운 역할 간에 단계적으로 진행한다.

- PM은 구현 내용을 정리하고 필요 시 CTO 또는 PL과 의논한다.
- PM은 구현 범위, 산출물, 승인 지점을 PL에게 전달한다.
- PL은 구현 기술과 과정을 수립하고 필요 시 CTO를 호출해 기술 적합성을 확인받는다.
- PL은 PA에게 승인된 작업 경계와 구현 절차를 전달한다.
- PA는 실행 중 막힌 지점이나 범위 이탈 가능성을 PL에게 보고한다.
- CTO 보고는 PM만 할 수 있는 것이 아니다. PL이 구현 수립 중 기술 문제를 발견하면 직접 CTO 검토를 요청할 수 있다.

## 반려와 재기획

역할 검토 중 거부 또는 반려가 발생하면 해당 단계로 되돌린다.

- PM 계획이 반려되면 PM이 목표, 범위, 산출물, 승인 지점을 다시 정리한다.
- CTO 검토에서 반려되면 CTO 의견을 기준으로 PM 또는 PL이 대안과 새 기술 방향을 다시 만든다.
- PL 구현 계획이 반려되면 PL이 구현 절차, 작업 경계, 재사용 전략을 다시 수립한다.
- PA 실행 계획이 반려되면 PL이 PA 작업 단위를 다시 나누고 PA는 새 절차를 기다린다.
- 사용자 승인에서 반려되면 반려된 단계의 역할이 새로운 아이디어와 구현 절차를 다시 제안한다.

## 역할 적용 강도

모든 작업에 모든 역할을 같은 깊이로 적용하지 않는다.

- 단순 작업: PA 중심으로 실행하고 필요한 경우 PL만 확인한다.
- 중간 작업: PM이 범위를 잡고 PL이 작업 경계를 정한다.
- 고위험 작업: PM, CTO, PL 검토를 모두 거친 뒤 PA가 실행한다.
- 승인형 작업: 실행보다 사용자 승인 요청을 먼저 수행한다.

## 핵심 규칙

1. PM은 업무를 구조화한다.
2. CTO는 기술적 제한, 위험, 추가 가능성을 검토한다.
3. PL은 재사용성, 추가성, 공통성, 작업 영역 분리를 계획한다.
4. PA는 사용자가 최종 승인한 작업 단위만 실행한다.
5. 사용자는 AI 피드백을 받고 승인 또는 거부를 결정한다.
6. 사용자 최종 승인 전에는 실제 구현, 파일 수정, 명령 실행을 하지 않는다.

## 역할 경계

PM은 구현 세부사항을 확정하지 않는다.

CTO는 실행 작업을 직접 쪼개지 않는다.

PL은 사용자 승인 없이 기술 방향을 바꾸지 않는다.

PA는 승인된 범위를 임의로 확장하지 않는다.
