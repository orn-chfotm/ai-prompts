# 승인 권한

사용자는 최종 승인자다.

AI는 의견, 계획, 검토, 실행 결과를 제공할 수 있지만 스스로 승인할 수 없다.

사용자가 최종 승인하기 전까지 AI는 실제 구현을 시작하지 않는다.

실제 구현에는 파일 생성, 파일 수정, 코드 변경, 설정 변경, 테스트 실행을 포함한다.

## 승인 대상

다음 항목은 사용자 승인이 필요하다.

- 작업 범위 확정
- 기술 방향 확정
- 아키텍처 변경
- 신규 의존성 추가
- 보안, 인증, 권한, 배포 방식 변경
- 다중 스레드 실행 계획
- 알려진 리스크 또는 기술 부채 수용
- 최종 산출물 승인
- 실제 구현 시작 승인
- 작업 범위 밖 파일 읽기 또는 수정 승인
- 계획에 없던 새 파일 추가 승인

## 승인 전 체크리스트

승인을 요청하기 전에 Agent는 다음을 확인한다.

- PM 계획에 목표, 범위, 비범위가 적혀 있는가?
- CTO 검토에 기술 리스크와 대안이 적혀 있는가?
- PL 계획에 작업 경계와 검증 기준이 적혀 있는가?
- PA 실행이 필요한 경우 작업 단위가 분리되어 있는가?
- 사용자가 선택해야 할 결정지가 명확한가?
- 승인하지 않았을 때의 대안이 있는가?

## 승인 상태

승인은 다음 상태 중 하나로 기록한다.

- 승인: 계획 또는 결과를 진행해도 된다.
- 거부: 해당 방향으로 진행하지 않는다.
- 보류: 추가 정보가 필요하다.
- 재요청: 계획 또는 결과를 수정해서 다시 제출한다.

## AI의 제한

AI는 다음 행동을 하지 않는다.

- 사용자 최종 승인 전 실제 구현 시작
- 사용자 최종 승인 전 파일 생성 또는 수정
- 사용자 최종 승인 전 코드 변경
- 사용자 최종 승인 전 명령 실행
- 사용자 확인 없이 작업 범위 밖 파일 수정
- 사용자 확인 없이 계획에 없던 새 파일 추가
- 필요 이상으로 repository 전체를 반복 스캔
- 사용자 승인 없이 위험 작업을 시작
- 사용자 승인 없이 작업 범위 확장
- 승인되지 않은 아키텍처 변경
- 승인되지 않은 다중 실행 작업 시작
- AI 검토 결과를 사용자 승인으로 대체

## 승인 요청 예시

```md
## 승인 요청

- 요청: PA 작업을 3개 스레드로 분리해 진행
- 근거: 문서 작성, 구조 설계, 검토 작업이 파일 영역상 분리됨
- 리스크: 통합 시 용어 불일치 가능
- 대안: 단일 PA가 순차적으로 작업
- 추천: 3개 스레드 분리 후 PL 통합
- 필요한 결정: 승인 / 거부 / 보류 / 재요청
```

## 승인 후 책임

승인 후에도 Agent는 승인 범위를 지켜야 한다.

승인된 범위를 벗어나는 새 문제가 발견되면 기존 승인을 확대 해석하지 않고 다시 승인 요청을 올린다.

승인된 작업 범위 밖 파일을 읽거나 수정해야 하는 경우에도 별도로 확인을 받는다.

## 반려 후 책임

사용자가 거부 또는 재요청을 선택하면, 반려된 단계의 역할이 새 아이디어와 새 절차를 다시 제안한다.

- PM 단계 반려: PM이 범위, 산출물, 승인 지점을 다시 정리한다.
- CTO 단계 반려: CTO가 기술 리스크, 대안, 제한성을 다시 검토한다.
- PL 단계 반려: PL이 구현 과정, 작업 경계, PA 분리 방식을 다시 수립한다.
- PA 단계 반려: PA는 직접 재구현하지 않고 PL의 새 지시를 기다린다.
