# 출력 형식

복잡한 작업은 다음 순서로 보고한다.

## 표준 보고 순서

1. PM 계획
2. CTO 검토
3. PL 개발 계획
4. PA 작업 분리
5. 사용자 승인 요청
6. 실행 결과
7. 검증 결과
8. 남은 리스크와 후속 작업

## 짧은 작업

작업이 작고 위험이 낮으면 모든 역할을 길게 출력하지 않는다.

필요한 역할만 사용하고, 생략한 이유를 짧게 적는다.

```md
## 간소화 보고

- PM: 범위가 단일 문서 수정으로 제한됨
- CTO: 기술 위험 없음
- PL: 작업 분리 불필요
- PA: 문서 수정 완료
```

## 승인 요청 형식

```md
## 사용자 승인 요청

- 결정할 내용:
- 추천 방향:
- 대안:
- 리스크:
- 승인 후 실행할 작업:
```

## 완료 보고 형식

```md
## 완료 보고

- 완료한 작업:
- 변경된 파일:
- 검증 방법:
- 남은 이슈:
```

## 보고 품질 기준

보고는 짧더라도 다음 조건을 만족해야 한다.

- 사용자가 무엇을 결정해야 하는지 보인다.
- 완료한 일과 하지 않은 일이 구분된다.
- 검증 여부가 명확하다.
- 실패, 보류, 미확인 항목을 숨기지 않는다.
- 다음 단계가 있으면 책임 역할이 드러난다.

## 역할별 최소 보고

```md
## PM 최소 보고

- 목표:
- 범위:
- 승인 필요 지점:
```

```md
## CTO 최소 보고

- 기술 방향:
- 리스크:
- 사용자 결정 필요 여부:
```

```md
## PL 최소 보고

- 작업 경계:
- PA 분리 여부:
- 검증 기준:
```

```md
## PA 최소 보고

- 변경 내용:
- 검증 결과:
- PL 확인 필요 사항:
```
