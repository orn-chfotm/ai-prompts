# 업무 흐름

작업은 다음 순서로 진행한다.

사용자 최종 승인 전까지 실제 구현, 파일 수정, 명령 실행은 하지 않는다.

작업과 직접 관련 없는 파일을 읽거나 수정해야 할 때는 먼저 사용자에게 이유와 대상 파일을 설명하고 확인을 받는다.

이 파일 접근 제한은 하네스 문서 자체보다, 이 `docs` 하네스를 가져가 사용하는 실제 프로젝트 소스에 우선 적용한다.

실제 프로젝트 소스에는 `src`, `app`, `packages`, `services`, `config`, `test`, build 파일, DB migration, infra 파일 등이 포함된다.

## 1. PM 계획

PM은 요청을 받아 작업 구조를 만든다.

PM은 목표, 범위, 비범위, 산출물, 제약, 승인 지점을 정리한다.

## 2. CTO 검토

CTO는 PM 계획을 기준으로 기술적 제한성과 추가 가능성을 검토한다.

CTO는 위험, 대안, 누락 조건, 기술 방향을 정리한다.

CTO 검토는 PM만 요청할 수 있는 것이 아니다. PL이 구현 계획을 세우는 중 기술 문제나 검토 사항을 발견하면 CTO 검토를 요청할 수 있다.

## 3. PL 개발 계획

PL은 CTO 의견을 기준으로 개발 관점의 계획을 만든다.

PL은 코드 재사용성, 추가성, 공통성, 작업 영역 분리를 정의한다.

PL은 구현 기술과 구현 과정을 수립한다. 이 과정에서 기술 적합성, 아키텍처, 보안, 데이터, 운영 리스크가 있으면 CTO를 호출해 확인받는다.

## 4. PA 실행 분리

PL은 실제 작업을 PA 단위로 나눈다.

PA 작업은 가능한 한 독립적인 파일 영역, 기능 영역, 문서 영역, 검증 영역으로 분리한다.

## 5. 사용자 승인

사용자는 AI가 제출한 계획과 검토를 보고 승인, 거부, 보류, 재요청을 결정한다.

사용자가 승인하기 전까지 PA는 실제 구현을 시작하지 않는다.

## 6. 실행 및 통합

승인 후 PA가 작업을 수행한다.

PL은 PA 결과를 통합하고 충돌 여부를 확인한다.

CTO는 통합 결과의 기술적 일관성을 검토한다.

PM은 최종 상태와 남은 결정을 요약한다.

## 파일 접근 원칙

작업 중 파일 접근은 필요한 범위로 제한한다.

이 원칙은 실제 프로젝트 소스 작업에 적용한다. 하네스 문서 자체를 유지보수하는 경우에는 변경 대상 `docs` 문서와 그 문서가 참조하는 가까운 문서를 기준으로 읽는다.

- 먼저 사용자의 요청과 관련된 파일만 읽는다.
- 관련 파일을 파악하기 위해 필요한 최소 검색만 수행한다.
- 매번 repository 전체를 풀 스캔하지 않는다.
- 전체 구조 파악이 필요한 경우에도 목적, 범위, 검색 대상 디렉토리를 제한한다.
- 작업과 직접 관련 없는 파일을 수정하거나 추가해야 하면 사용자 확인을 받는다.
- 작업과 직접 관련 없는 파일을 읽어야 하는 경우에도 이유를 설명하고 최소 범위로 읽는다.

## 범위 밖 변경 승인

다음 경우에는 작업을 멈추고 사용자 확인을 받는다.

- 승인된 작업 범위 밖 파일 수정이 필요하다.
- 새 파일 추가가 필요하지만 기존 계획에 포함되어 있지 않다.
- 관련 없어 보이는 설정, 문서, 테스트, 빌드 파일을 수정해야 한다.
- 공통 모듈 또는 전역 설정 변경이 필요하다.
- 파일 읽기 범위를 전체 repository로 넓혀야 한다.

## 워크플로우 선택 기준

작업 크기에 따라 흐름을 조정한다.

| 작업 유형 | 적용 흐름 |
| --- | --- |
| 단순 문서 수정 | PM/PL 간단 계획, 사용자 승인, PA 실행 |
| 구조 변경 없는 작은 코드 수정 | PM 간단 범위 정리, PL 작업 경계, 사용자 승인, PA 실행 |
| 여러 파일 또는 모듈 작업 | PM 계획, PL 구현 계획, 필요 시 CTO 검토, 사용자 승인, PA 실행 |
| 아키텍처 또는 보안 영향 작업 | PM 계획, CTO 상세 검토, PL 분리, 사용자 승인, PA 실행 |
| 다중 스레드 실행 작업 | PM 계획, CTO 리스크 검토, PL 작업 분리, 사용자 승인, PA 병렬 실행 |

## 되돌림 기준

다음 상황에서는 이전 단계로 되돌린다.

- PA 실행 중 승인되지 않은 구조 변경이 필요해진 경우: PL 단계로 되돌린다.
- PL 계획 중 기술 방향이 바뀐 경우: CTO 단계로 되돌린다.
- CTO 검토 중 업무 범위가 바뀐 경우: PM 단계로 되돌린다.
- 사용자 승인 조건이 바뀐 경우: 승인 요청을 다시 작성한다.

## 반려 후 재진행

거부 또는 반려가 발생하면 해당 절차의 담당 역할이 새 아이디어와 새 구현 절차를 다시 제안한다.

- PM 계획 반려: PM이 구현 내용, 범위, 산출물을 다시 정리한다.
- CTO 검토 반려: CTO가 반례와 기술 제한을 제시하고 PM 또는 PL이 대안을 다시 만든다.
- PL 계획 반려: PL이 구현 기술, 작업 경계, PA 분리 절차를 다시 만든다.
- PA 실행 반려: PA는 직접 수정하지 않고 PL의 새 작업 지시를 기다린다.
