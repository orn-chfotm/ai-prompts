# 작업 분리 기준

PA 작업은 충돌 가능성이 낮고 결과 통합이 명확할 때만 분리한다.

## 분리 가능한 기준

다음 기준 중 하나 이상을 만족하면 분리할 수 있다.

- 파일 영역이 다르다.
- 기능 영역이 다르다.
- 문서 작성과 코드 구현이 분리된다.
- 테스트 작성과 구현이 분리된다.
- 조사, 설계, 실행, 검토가 독립적으로 진행될 수 있다.

## 분리하면 안 되는 경우

다음 경우에는 여러 PA로 나누지 않는다.

- 같은 작은 파일을 동시에 수정한다.
- 작업 순서 의존성이 강하다.
- 설계가 확정되지 않았다.
- 승인되지 않은 기술 방향이 포함된다.
- 통합 비용이 실행 비용보다 크다.

## PA 작업 예시

```md
## PA 작업 분리

- PA-1: 공통 문서 초안 작성
- PA-2: 역할별 문서 작성
- PA-3: Agent별 진입점 문서 작성
- PA-4: 전체 용어와 링크 검토
```

## 통합 기준

PL은 PA 결과를 통합할 때 다음을 확인한다.

- 용어 일관성
- 중복 규칙 여부
- 역할 경계 침범 여부
- 승인 게이트 누락 여부
- Codex와 Claude 양쪽에서 해석 가능한지 여부

## 병렬 실행 리스크

여러 PA 작업은 속도를 높일 수 있지만 통합 리스크를 만든다.

PL은 병렬 실행 전에 다음을 확인한다.

- 같은 파일을 동시에 수정하지 않는가?
- 같은 용어를 서로 다르게 정의하지 않는가?
- 한 PA의 결과가 다른 PA의 전제 조건이 아닌가?
- 통합 책임자가 명확한가?
- 실패한 PA 작업을 제외해도 나머지 결과를 사용할 수 있는가?

## 작업 카드 형식

PA 작업은 다음 형식으로 분리한다.

```md
## PA 작업 카드

- 작업 ID:
- 담당 역할:
- 목표:
- 수정 가능 영역:
- 수정 금지 영역:
- 입력 문서:
- 읽기 대상 파일:
- 읽기 제한:
- 산출물:
- 검증 기준:
- PL에게 보고할 항목:
```

## 에스컬레이션 기준

PA는 작업 중 다음 상황을 발견하면 PL에게 올린다.

- 수정 가능 영역 밖의 변경이 필요하다.
- 수정 가능 영역 밖의 파일을 읽거나 분석해야 한다.
- 기술 방향이 CTO 검토와 다르다.
- 사용자 승인이 필요한 새 결정이 생긴다.
- 다른 PA 작업과 충돌한다.

## 파일 범위 제한

PL은 PA 작업을 나눌 때 읽기 대상과 수정 가능 영역을 함께 지정한다.

PA는 지정된 읽기 대상과 수정 가능 영역을 우선한다.

지정되지 않은 파일을 읽거나 수정해야 하면 PL에게 보고하고, 범위 밖 변경이면 사용자 확인을 받는다.
