# 리뷰 절차

리뷰는 변경의 정확성, 범위 준수, 기술 안정성, 검증 가능성을 확인한다.

## 역할별 리뷰

- PA: 자신이 수행한 변경과 검증 결과를 보고한다.
- PL: PA가 생산한 코드의 1차 리뷰를 수행하고, 작업 경계, 통합 충돌, 재사용성, 공통성 위반 여부를 확인한다.
- CTO: 기술 방향, 보안, 성능, 데이터, 운영 리스크를 확인한다.
- PM: 요청 목표와 산출물이 맞는지 확인한다.
- 사용자: 최종 승인, 거부, 보류, 재요청을 결정한다.

## 리뷰 기준

- 승인된 범위 안에서 작업했는가?
- 불필요한 변경이 섞이지 않았는가?
- 테스트 또는 검증 결과가 명확한가?
- 남은 리스크가 사용자에게 보고되었는가?

## PL 1차 코드 리뷰 기준

PL은 PA 산출 코드를 통합하기 전에 다음을 확인한다.

- PA가 승인된 작업 범위와 파일 경계를 지켰는가?
- 코드가 기존 구조, naming, 스타일, 테스트 패턴과 일관적인가?
- 재사용 가능한 기존 코드나 공통 모듈을 무시하지 않았는가?
- 작업 단위가 리뷰 가능한 크기와 목적을 유지하는가?
- 테스트 실패, 미검증 항목, blocker가 숨겨져 있지 않은가?
- error, critical, improvement로 기록할 항목이 있는가?

PL 리뷰에서 범위 밖 변경, 아키텍처 판단, 보안 또는 데이터 리스크가 발견되면 CTO 검토로 에스컬레이션한다.

## 리뷰 기록

리뷰 중 다음 항목이 발견되면 `04-review-logging.md` 기준에 따라 기록한다.

- 에러: 실제 실패, 오류, 재현 가능한 문제
- 크리티컬: 보안, 데이터 손실, 배포 실패, 장애 가능성이 큰 문제
- 개선 사항: 당장 실패는 아니지만 품질, 유지보수성, 테스트 안정성을 높이는 항목

기록 위치는 이 하네스 문서 submodule 내부가 아니라, 하네스를 가져간 실제 프로젝트 내부여야 한다.
