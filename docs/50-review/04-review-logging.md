# 리뷰 로깅

이 문서는 리뷰 중 발견한 에러, 크리티컬, 개선 사항을 어디에 어떻게 기록할지 정의한다.

중요한 원칙은 기록 위치다.

리뷰 로그는 `docs` 하네스 submodule 내부에 남기지 않는다.

리뷰 로그는 이 하네스를 가져간 실제 프로젝트의 작업 기록 영역에 남긴다.

## 기록 대상

다음 항목은 로그로 남긴다.

- 에러: 명령 실패, 빌드 실패, 테스트 실패, 런타임 오류, 재현 가능한 동작 오류
- 크리티컬: 보안 위험, 데이터 손실, 권한 문제, 배포 실패, 운영 장애 가능성이 큰 문제
- 개선 사항: 품질, 유지보수성, 테스트 안정성, 문서 명확성을 높일 수 있는 항목
- AI 설계 기록: AI 하네스, Agent 구조, 지침 파일, 역할 분리, 실행 흐름에 대한 설계 판단
- AI 피드백 기록: 사용자가 AI의 응답 방식, 검토 품질, 작업 방식에 대해 남긴 피드백
- AI 개선점 기록: 이후 AI 작업 품질을 높이기 위해 반영해야 할 개선 사항

## 기록 위치

프로젝트에 이미 리뷰 또는 작업 로그 위치가 있으면 그 위치를 따른다.

프로젝트에 정해진 위치가 없으면 다음 경로를 기본값으로 사용한다.

```text
.ai/reviews/
```

프로젝트에 `.ai/reviews/README.md`가 있으면 그 파일의 디렉토리 분류와 파일 네이밍 규칙을 우선한다.

기본 디렉토리 분류는 다음과 같다.

| 디렉토리 | 용도 |
|---|---|
| `.ai/reviews/audit/` | 코드 오류, critical, error, improvement 리뷰 |
| `.ai/reviews/feature/` | 미구현 기능 갭 분석, 기능 추가 계획 |
| `.ai/reviews/design/` | AI 하네스, Agent 구조, 지침 파일, 역할 분리 설계 기록 |
| `.ai/reviews/feedback/` | 사용자가 남긴 AI 응답 방식, 검토 품질, 작업 방식 피드백 |

사용자가 "AI의 설계", "AI 피드백", "AI 개선점", "기록으로 남겨줘"처럼 명시하면 일반 대화 답변과 별도로 `.ai/reviews/` 하위에 독립된 md 파일로 남긴다.

날짜와 slug를 사용할 때는 다음 형식을 권장한다.

```text
.ai/reviews/<category>/YYYY-MM-DD-{slug}.md
```

작업 단위별 파일을 사용할 때는 다음 형식을 권장한다.

```text
.ai/reviews/<category>/<task-id>.md
```

상태 표기는 다음 기준을 따른다.

| 상태 | 의미 |
|---|---|
| `[ ]` | 미처리 |
| `[x]` | 처리 완료 |
| `[-]` | 보류 또는 wontfix |

## submodule 사용 시 기준

이 하네스를 git submodule로 가져간 프로젝트에서는 다음 기준을 따른다.

- 하네스 submodule 내부 파일에는 프로젝트별 리뷰 로그를 남기지 않는다.
- 실제 프로젝트 repository의 문서 또는 로그 디렉토리에 기록한다.
- 프로젝트별 로그 위치가 있으면 그 위치를 우선한다.
- 로그 위치가 없으면 프로젝트 root 기준 `.ai/reviews/`를 제안한다.
- 실제 프로젝트가 하네스를 `docs/` 경로로 submodule import하는 경우 `docs/reviews/`는 사용하지 않는다.

## 로그 형식

```md
## Review Log

- 날짜:
- 작업:
- 작성 역할:
- 심각도: error | critical | improvement
- 분류:
- 위치:
- 내용:
- 재현 방법:
- 영향 범위:
- 제안 조치:
- 상태: open | resolved | deferred | rejected
- 사용자 승인 필요 여부:
```

## 심각도 기준

### error

실패가 발생했거나 재현 가능한 문제가 있는 경우다.

예시는 다음과 같다.

- 테스트 실패
- 빌드 실패
- 명령 실행 실패
- 요구사항과 다른 동작

### critical

사용자 승인 없이 넘어가면 안 되는 높은 위험이다.

예시는 다음과 같다.

- 보안 취약점
- 인증 또는 권한 우회 가능성
- 데이터 손실 가능성
- 배포 실패 가능성
- 운영 장애 가능성

### improvement

즉시 실패는 아니지만 품질을 높일 수 있는 개선 사항이다.

예시는 다음과 같다.

- 테스트 보강
- naming 개선
- 문서 명확화
- 중복 제거
- 유지보수성 개선

## 역할별 책임

- PA: 발견한 에러와 검증 실패를 숨기지 않고 기록 후보로 보고한다.
- PL: 기록 여부, 심각도, 작업 영향 범위를 판단한다.
- CTO: critical 항목의 기술 위험과 승인 필요 여부를 판단한다.
- PM: 사용자에게 결정이 필요한 항목을 요약한다.
- 사용자: critical 또는 범위 변경 항목을 승인, 거부, 보류, 재요청한다.

## 금지 사항

- 하네스 submodule 내부에 프로젝트별 review log를 누적하지 않는다.
- 실패한 테스트를 improvement로 낮춰 기록하지 않는다.
- critical 항목을 사용자 승인 없이 resolved 처리하지 않는다.
- 로그 없이 구두 보고만 하고 넘어가지 않는다.
